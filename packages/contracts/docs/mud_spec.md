# Hex-Shogun-Map: MUD データ層移行仕様書

本ドキュメントは、`hex-shogun-map` プロジェクトを完全オンチェーンの世界（MUD/ECSアーキテクチャ）における **「データ層（土地・環境・領地の備蓄）」として特化して構築** するための仕様書です。

> ※ 本リポジトリはデータ層として独立稼働し、人物情報の管理（Universal-Soul-Logic）や行動ルール（universal-province-engine）、およびインターフェースの実描画（sengoku-hex-viewer等）は別リポジトリで管理され、外部から本リポジトリのデータを参照・操作します。

---

## 1. プロジェクトスコープ（管理対象）

本プロジェクトがMUDのStore（データ層）として定義し、責任を負う領域は以下の通りです。

### Entity（実体）
MUDデータベースにおいて、属性（Component）を紐づける箱（ID）です。
*   **土地（ヘックス）**: 座標によって一意に識別される世界上の「区画」。既存JSONデータの `hex_id` 等がEntityの識別子となります。
*   **城郭・施設**: その土地に建設された建造物。

### Component（属性）
以下のコンポーネントを `mud.config.ts` で定義し、当該Entityに紐付けます。

1. **地理・環境系**
   *   `PositionComponent`: ヘックスの座標空間。内部的には `col`（列）と `row`（行）で管理されますが、これらは `lat`（緯度）と `lng`（経度）から直接計算・相互変換できる仕様となっています。このため、緯度経度情報を持つマップデータを利用すれば、**現実世界のGPS位置情報と完全に連動したAutonomous World**を標準で構築できます。
   *   `TerrainComponent`: 地形情報（`terrain_type`: 山、川、平野など）
   *   `EnvironmentComponent`: 標高(`elevation_m`)や災害フラグ
   *   `SubGridEntryRuleComponent`: 子グリッドへの進入時のルール（`RANDOM`, `SELECT` 等）

2. **領地の備蓄（資源・経済）系**
   *   `ProvinceProvisionComponent`: 領地の「蔵」に備蓄された兵糧。軍勢の遠征などに引き出されます。
   *   `TaxRevenueComponent`: 市場や開墾から得られた税収（内政・治水のための待機資金）。
   *   `TerritoryStatComponent`: 石高、人口、開発度などの潜在的な値。

## 2. 外部連携の仕様（動作認識）

本リポジトリ自体は振る舞いのロジックを持ちませんが、外部からの相互連携によって以下のように挙動（データの増減）が変化することを前提とします。

*   **所有権の移転（外部同期）**: 土地Entityの持ち主がいずれかのSoul（人物）に切り替わった場合、`ProvinceProvisionComponent` 等の保有資産の支配権は自動的にそのSoulの管理下へ移ります。
*   **イベントによる数値変動**: 外部ロジック層（`DevelopmentSystem`等のトランザクション）からコールされることで、本リポジトリの `TerritoryStatComponent`（石高）や `TaxRevenueComponent` が増減します（疎結合なバケツリレー）。

## 3. 実装タスクリスト

本リポジトリにおけるMUD化の開発タスクは以下の通りです。

### [フェーズ1] MUD基盤の立ち上げとテーブル定義
- [ ] MUDプロジェクト（Foundry環境）の初期化。
- [ ] `mud.config.ts` にて、上記の「地理・環境」および「領地の備蓄」コンポーネント群のスキーマ（テーブル）を定義する。

### [フェーズ2] 既存JSONデータからのマイグレーション（初期データ投入）
現在の `sengoku/` フォルダ等のJSONファイルに存在するデータを「天地創造のシード（種）データ」として活用します。
- [ ] 各地方のJSONデータ（`hex_id`, `col`, `row`, `terrain_type`, 標高など）を読み込む。
- [ ] `PostDeploy` スクリプトを作成し、JSONデータをパースしてMUDの各Entity（ヘックス等）と紐づくComponentの初期値としてコントラクトへ流し込む。

### [フェーズ3] モック連携テスト
- [ ] 本データのテーブルが正常に読み書きできるかを確認するシンプルなテストスクリプト(`forge test`)を作成。
- [ ] 外部ロジックによる「石高増加」「兵糧消費」等の挙動を想定したテストケースをSolidity内で記述・検証する。

## 4. マルチNamespaceアーキテクチャ（管理母体の分離とクローン展開）

本データ層はMUDの**「Namespace（名前空間）」**機能を利用し、外部のロジックや他チームからのアクセス権限を厳密に管理します。これにより、「1つの大地データ（Store）」を共有しながら、ルール（物理法則）の異なる無数の「クローン世界」や「パラレルワールド」を容易に構築することが可能になります。

### A. Namespaceの分割設計
本リポジトリがデプロイするMUD Storeは、以下の名前空間に限定されます。
*   **`namespace: "hex_map"`**
    *   **所有権**: 本プロジェクト（`hex-shogun-map` 開発管理チーム）
    *   **書き込み権限**: 本チームのみ。あるいは、ガバナンス（DAO）によって承認された特定システム（`universal-province-engine` 等）のみ。
    *   **読み取り権限**: パブリック（誰でも自由・制限なし）。

### B. 別世界（パラレルワールド・クローン）を作るのが劇的に簡単になる理由
別の開発チームやユーザーが「この土地データを使って自分のルール（例えば魔法の世界）で遊びたい」と考えた場合、ゲームエンジン全体をフォーク（丸ごとコピー）する必要はありません。以下の手順で**完全に独立したクローン別世界が簡単に作れます。**
1.  **既存のデータ層（Store）に相乗りする**: 本プロジェクトが提供するMUDの同じ世界（Worldコントラクト）に対して、自分用の新しい名前空間（例：`namespace: "sf_colony"`）を登録し、独自のロジックコントラクトを追加デプロイします。
2.  **フロントエンドの再利用（Skinの切り替え）**: `hex_map` の地形データ（共有インフラ）をそのまま無償で読み出し、自分たちのフロントエンドで「SF世界」としてUIを描画し、自分たちのロジックだけを走らせます。

> **【実証サンプルとしてのアルカディア】**
> 本リポジトリには、この「パラレルワールド・クローン展開」の強力さを証明するための初期の実証サンプルとして、日本の戦国時代だけでなく、**異世界ファンタジー「アルカディア」の地形データが併せて格納されています。**

## 5. MUD化におけるファイル・ディレクトリ構成案

**「各マップにつきデータ層（MUDプロジェクト）は2つ必要か？」という疑問への回答:**
結論として、**1つのMUDプロジェクト（1つのブロックチェーンWorld）の中に、複数のマップデータを同居させることが可能**です。例えば `mud.config.ts` で `sengoku` という名前空間と `arcadia` という名前空間を並列して定義することで、1つの同じブロックチェーン上に「2つの別次元の地球のデータベース」を展開できます。

これを踏まえ、フェーズ1以降の本格的なMUD移行において目指すべき「データ層としてのディレクトリ構成」のひな形は以下のようになります。

```text
hex-shogun-map/
├── README.md
├── mud_spec.md             # 本仕様書
├── packages/
│   ├── contracts/          # MUD スマートコントラクト（データ層本体）
│   │   ├── mud.config.ts   # Component（テーブル）と複数名前空間（sengoku, arcadia）の定義
│   │   ├── src/
│   │   │   └── systems/    # （※データ層に特化するため、基本的に参照系ロジックのみ）
│   │   ├── script/
│   │   │   └── PostDeploy.s.sol # JSONをパースして各マップの初期データを流し込むスクリプト
│   │   └── data/           # スマートコントラクトに初期投入するためのシードデータ置場 ★
│   │       ├── sengoku/    # 例: sengoku/data にある既存のJSONデータを移行配置
│   │       └── arcadia/    # 例: arcadia/data にある既存のJSONデータを移行配置
│   │
│   └── client/             # テスト用インターフェース層（MUD RECSとの統合モック）
│       ├── index.html      # 現在のポータルページ
│       ├── sengoku/        # 戦国UI等、現在の描画モジュールそのまま
│       ├── arcadia/        # 異世界UI等
│       └── src/
│           ├── mud/        # MUDブロックチェーンと同期してローカルRECSに保存する同期アダプタ
│           └── app.js      # 現在のキャンバス描画ロジック（※RECSからデータをもらうよう改修）
```
現在フロントエンドの中に散らばっている `sengoku/data` や `arcadia/data` の内容は、MUD化の際に上記の `contracts/data/` 直下へシードデータとして集約し、スクリプトで両方の世界を一気にオンチェーンに流し込む設計とします。
