# ガス代削減と仮想テスト（Simulation）の設計指針

本ドキュメントは、ブロックチェーン（MUD）への書き込みコスト（ガス代）を最小限に抑えつつ、GitHub Pagesなどの静的環境で「仮想動作」させるための設計指針です。

---

## 1. ゼロコストの仮想テスト（Simulation Mode）

GitHub Pagesやローカル環境で「ガス代ゼロ」でテスト・検証を行うため、**「データアクセス層（DAO）」**を仮想化します。

### A. 仕組み: Mock同期アダプタ
*   **通常モード**: `Viem` を通じてブロックチェーンの実ノードと通信。
*   **仮想モード（Simulation）**: `fetch()` でJSONファイルを自前で読み込み、MUDの `RECS`（メモリ内DB）に直接書き込む。

### B. 書き込み（Transaction）の仮想化
JavaScriptのレイヤーで「コントラクト呼び出し」をインターセプトし、成功したと仮定してローカルの `RECS` を書き換えます。これにより、**UIの挙動は本物と全く同じでありながら、一切のガス代を発生させずに検証可能**です。

---

## 2. ガス代を削減するためのアーキテクチャ（Gas-Saving Design）

実際にブロックチェーンへ書き込む際のコストを抑えるための3つの戦略です。

### ① 更新の遅延と集約（Lazy & Batch Updates）
*   **全セル更新を避ける**: 例えば「毎秒兵糧を消費する」という計算をオンチェーンで行うとガス代が膨大になります。
*   **解法**: 数値の計算（Delta）はオフチェーン（Client/Indexer）で行わせ、「精算（Settle）」が必要なタイミング（例：部隊が城に帰還した時）だけ、最終的な結果を一括でオンチェーンに書き込みます。

### ② 差分更新（Delta Updates）
*   MUDの `Store` は、変更されたフィールドのみを書き換える機能があります。初期デプロイ後は、変化のない「地形種別」などは読み取り専用とし、変動の激しい「兵糧」や「石高」のみを最小限の関数で更新します。

### ③ 名前空間による「書き込み制限」の活用
*   全てのSystemをWorldに登録するのではなく、本データの管理母体（hex-shogun-map）が認めた特定のロジック層（System）のみに書き込み権限を限定することで、無駄な状態遷移（トランザクション）の発生を抑制します。

---

## 3. 実装のロードマップ

| フェーズ | 環境 | ガス代 | 内容 |
| :--- | :--- | :--- | :--- |
| **Step 1: 仮想検証** | GitHub Pages | **0円** | JSONデータをMock同期してUI/UXを検証。 |
| **Step 2: ローカル検証** | Anvil (Local) | **0円** | 自分のPC内で仮想通貨（テスト用）を使って動作確認。 |
| **Step 3: テストネット** | Sepolia 等 | **0円** | 無料のテスト用通貨を使ってネットワーク遅延などを検証。 |
| **Step 4: 本番稼働** | Mainnet | 有料 | 最小限のガス代で、不変のデータをブロックチェーンへ刻印。 |

> [!TIP]
> テストケース(`DataLayerTest.t.sol`)においても、`vm.prank` や `vm.warp` といった Foundry の機能を使うことで、実環境での時間経過や権限を「ガス代ゼロ」で完璧に再現できます。
